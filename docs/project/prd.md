# PUDIWIND 亚马逊自动化运营系统 PRD

## 1. 产品目标
本系统是一套仅供 Jim 与运营团队内部使用的亚马逊运营辅助系统，目标是在安全可控前提下，把认证、数据看板、Agent 协作、审批、系统管理等能力沉淀为一个可持续迭代的内部工作平台。

当前阶段坚持两条主线：
1. 用 Mock-first 方式先验证核心交互与业务链路。
2. 所有 AI 产出与自动化动作必须经过人工审批或人工确认后再固化或执行。

## 2. 目标用户与角色边界
### Boss
- 可访问全部页面与系统管理能力
- 可使用 Boss 专属能力，如品牌规划、审计、知识库审核、系统配置
- 负责审批与最终决策

### Operator
- 可访问日常运营页面与通用 Agent
- 不可进入系统管理、Boss 专属 Agent、知识库审核等高权限区域
- 负责日常执行与数据查看

## 3. Phase 1 范围
当前仓库处于 Phase 1，以 Mock 数据与可验证流程为主，目标是先把基础工作台跑通。

### 已在范围内
- JWT 登录、刷新、当前用户信息
- 基于角色的页面保护与接口鉴权
- 数据大盘、广告看板、广告管理
- 订单列表、退货列表
- Agent 列表与聊天入口
- 审批中心、知识库审核
- 系统管理：用户管理、Agent 配置、API 密钥状态、费用监控
- APScheduler 后端调度接口
- PostgreSQL / pgvector 数据模型基础
- Docker 部署基线

### 明确不在当前范围内
- 真实 Amazon 写入链路
- 前端保存 API key 或直接调用敏感服务
- 无审批的自动执行闭环
- 退货详情页与退货详情接口
- 前端创建/编辑 Cron 任务能力

## 4. 当前已实现能力
### 4.1 认证与安全
- 登录、刷新、`/api/auth/me` 已实现
- 非公开接口统一经过 JWT 中间件校验
- 前端通过 Axios 拦截器附加 Bearer token
- API key 仅在服务端环境变量中管理，前端只展示状态

### 4.2 运营工作台
- 首页数据大盘已完成第一轮口径收口
- 广告大盘与广告管理已具备 Mock 数据与主流程
- 订单与退货列表已完成字段归一化与契约测试
- 订单详情已完成前端归一化映射

### 4.3 Agent 与审批
- Agent 列表、Agent 聊天、审批中心、知识库审核已具备基础页面与 API
- 审批流仍是平台核心安全边界之一，不能绕过

### 4.4 系统管理
- 用户管理、Agent 配置、API 密钥状态、费用监控已具备独立页面或能力
- 计划任务后端接口已存在，但独立前端页面尚未接入完成

## 5. 当前缺口
### 第一优先级
- `/system/schedules` 仍为占位页，尚未接入真实计划任务页面
- 该能力已有后端接口与调度配置基础，适合作为当前最小闭环补齐项

### 第二优先级
- 退货方向只有列表页与列表接口，尚无 detail endpoint 与详情页
- 聊天记录持久化与飞书端实际使用体验仍有优化空间

## 6. 产品规则
- 所有敏感能力默认以服务端为可信边界，不能信任前端输入
- 所有自动化动作必须可审计、可追溯、可回滚
- 所有 Agent 产出默认先进入人工确认链路，而不是直接执行
- 新功能优先遵循小步迭代、复用现有 API、避免同时改动多个无关模块

## 7. 下一阶段优先级
1. 补齐计划任务独立页面，打通 `/system/schedules`
2. 继续收口系统管理与工作台缺口
3. 视范围与收益再评估退货详情链路
4. 在 Mock-first 验证稳定后，再讨论真实 Amazon API 接入顺序
