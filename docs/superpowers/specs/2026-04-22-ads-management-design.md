# 广告管理页设计稿

日期：2026-04-22

## 背景与目标

当前 Web 端除广告管理页外，其余核心页面已基本稳定。广告管理页需要参考赛狐 ERP 的“广告管理”列表总页逻辑进行重构，但只落地符合本项目当前运营流程的必要能力。

本设计基于以下现状整理：
- 当前主页面已具备基础筛选与 tab 骨架：`src/frontend/src/pages/AdManagement.tsx:19`、`src/frontend/src/pages/AdManagement.tsx:40`
- 当前后端已提供 8 个管理 tab 对应的 Mock API：`src/api/ads.py:125`
- 当前已有广告活动详情页基础：`src/frontend/src/pages/ad-management/CampaignDetail.tsx:153`
- 参考页真实结构已通过浏览器验证，核心对象视图、左侧组合筛选、顶部全局工具栏、主表格和分页链路已确认

本次设计目标不是 1:1 复刻赛狐完整广告系统，而是构建一套适合当前 Mock 阶段、可扩展、可测试的“广告对象管理工作台”。

## 范围

### 第一版纳入范围
- 广告组合
- 广告活动
- 广告组
- 广告产品
- 投放
- 搜索词
- 否定投放
- 广告日志
- 左侧广告组合筛选
- 顶部全局筛选
- 主表格分页、空态、错误态、加载态
- 从广告组合到广告活动的下钻
- 从广告活动到活动详情页的跳转

### 明确排除范围
- 广告位
- AMC

- 高级批量动作业务闭环
- 与当前运营流程无关的扩展模块

## 设计原则

- **KISS**：保留单页工作台模型，不引入多页面并行模式
- **YAGNI**：仅实现当前运营流程真实需要的对象视图和交互
- **DRY**：分页、加载、错误处理、表格容器逻辑统一复用
- **SOLID / SRP**：左侧筛选、顶部工具栏、对象 tab、主表格分别承担单一职责
- **精确改动**：优先升级现有 `AdManagement.tsx` 和既有 ads API，不做无关重构

## 页面定位

广告管理页应被定义为“广告对象管理工作台”，而不是单纯的表格页。

它需要统一解决三类问题：
1. 在同一页面内切换不同广告对象视角
2. 保持跨对象视图的一致筛选体验
3. 提供稳定的下钻链路，从总览进入更细层级

## 页面分区

### 1. 左侧筛选面板
职责：承担广告组合维度的一级筛选入口。

包含内容：
- 广告组合搜索框
- 广告组合列表
- 全部广告组合重置入口
- 可见 / 隐藏 / 对比视图入口
- 单个组合的“仅筛选此项”动作
- 当前已选组合状态反馈

行为约束：
- 左侧搜索只过滤左侧列表显示，不直接搜索主表
- 选中一个或多个组合后，主表仅展示相关数据
- “仅筛选此项”覆盖当前选中集合
- “全部广告组合”清空组合筛选并恢复全量结果

### 2. 顶部工具栏
职责：承载所有全局查询条件和全局级动作。

建议包含：
- 店铺选择
- 广告类型切换
- 服务状态筛选
- 日期范围筛选
- 顶部关键词搜索框
- 同步按钮
- 重置按钮
- 第一版必要的快捷操作入口

行为约束：
- 顶部条件对当前 tab 生效，但状态由页面共享
- 任一条件变更后重置分页到第一页
- 重置操作恢复默认筛选并刷新当前 tab 数据

### 3. 对象 Tab 区
职责：切换广告对象视图。

第一版 tab 固定为：
- 广告组合
- 广告活动
- 广告组
- 广告产品
- 投放
- 搜索词
- 否定投放
- 广告日志

行为约束：
- tab 只切对象，不切整套页面交互模型
- 切换后保留全局筛选：店铺、广告类型、服务状态、日期范围、组合筛选
- 切换后重置：分页、排序、顶部关键词搜索

### 4. 主表格工作区
职责：展示当前对象数据，并承担工作台主操作。

建议包含：
- 当前 tab 对应的数据表格
- 批量选择区
- 自定义列入口（第一版可仅做结构）
- 行级操作入口
- 分页器
- 空态 / 错误态 / 加载态
- 可选汇总行

行为约束：
- 表格字段随 tab 切换
- 表格容器逻辑统一，字段与接口通过 schema 驱动
- 第一版优先保证分页、错误态和下钻稳定性

## 共享状态模型

建议广告管理页统一维护以下查询状态：

- `activeTab`
- `shopId`
- `adType`
- `serviceStatus`
- `dateRange`
- `selectedPortfolioIds`
- `portfolioSearch`
- `keyword`
- `page`
- `pageSize`
- `sortBy`
- `sortOrder`

状态持有原则：
- 页面层持有共享状态
- 子组件只消费与回写自己相关的字段
- 避免在多个子组件中重复维护分页、搜索和排序状态

## 数据流

### 初始化
- 页面加载时拉取广告组合树
- 默认 tab 建议从 `campaign` 调整为 `portfolio`，更符合广告管理总览入口心智
- 默认分页为第 1 页，每页 20 条

### 筛选变更
任一筛选项变更时：
1. 更新共享状态
2. 将 `page` 重置为 1
3. 重新请求当前 tab 数据

### Tab 切换
切换 tab 时：
1. 保留全局筛选上下文
2. 重置分页和排序
3. 清空当前顶层关键词搜索
4. 加载新 tab 的 schema 和数据

### 行级下钻
- 点击广告组合名称：进入“广告活动”视图，并自动带入该组合筛选
- 点击广告活动数量：进入“广告活动”视图，并自动带入该组合筛选
- 点击广告活动名称：进入 `CampaignDetail` 详情页

## 表格 Schema 设计

建议以 schema 驱动每个 tab，而不是在渲染层堆叠大量条件分支。

建议结构：

```ts
 type AdsTabSchema = {
   endpoint: string
   buildParams: (query: AdsQueryState) => Record<string, unknown>
   columns: Column<Record<string, unknown>>[]
   getRowKey: (row: Record<string, unknown>) => string
   enableBatchSelection: boolean
   emptyText: string
 }
```

并维护：

```ts
const TAB_SCHEMAS: Record<TabKey, AdsTabSchema> = { ... }
```

Schema 需要承担的职责：
- 指定当前 tab 请求哪个接口
- 将共享状态转换为接口参数
- 声明表格字段和列格式化方式
- 声明行级操作入口
- 声明空态文案与是否支持批量选择

这样可以实现：
- 表格容器逻辑只写一次
- 新增或调整 tab 时主要修改 schema
- 减少 `if/else` 分支导致的维护成本

## API 映射策略

后端已有 8 个管理对象 endpoint，可作为第一版基础：
- `GET /api/ads/portfolios`
- `GET /api/ads/campaigns`
- `GET /api/ads/ad_groups`
- `GET /api/ads/ad_products`
- `GET /api/ads/targeting`
- `GET /api/ads/search_terms`
- `GET /api/ads/negative_targeting`
- `GET /api/ads/logs`

后续实现建议逐步统一这些接口的筛选语义，使其能更自然支持：
- `portfolio_id`
- `ad_type`
- `service_status`
- `keyword`
- `date_range`
- `page`
- `page_size`

第一版不要求所有 endpoint 一次性完全统一，但应保证前端 query model 的方向一致。

## 错误处理与状态反馈

### 加载态
- 左侧广告组合树和主表格分开 loading
- 切 tab 或变更筛选时，只刷新主表区域，不整页闪烁

### 空态
- 有筛选条件无结果：显示“当前筛选下暂无数据”
- 无任何数据：显示“暂无广告数据”

### 错误态
- 请求失败时提供可重试提示
- 不清空当前筛选条件
- 保留当前页面上下文，避免用户重复输入

## URL 状态同步

建议第一版将以下关键状态同步到 URL search params：
- `tab`
- `adType`
- `shop`
- `portfolio`
- `page`
- `keyword`

目的：
- 刷新后恢复当前视图
- 方便运营复制链接沟通
- 便于手动回归与问题复现

日期范围和部分临时 UI 状态第一版可不完全同步，避免 URL 过重。

## 测试策略

### 前端契约测试
重点保护：
- tab 集合固定为 8 个业务相关视图
- 不包含广告位和 AMC
- 默认 tab 与主布局结构正确
- 左侧筛选面板关键入口存在
- 下钻与详情跳转入口存在

### API 契约测试
重点保护：
- 8 个 tab endpoint 的返回结构稳定
- 保持 `{items, total_count, summary_row}` 契约
- 关键筛选参数和分页参数结构稳定
- drill-down endpoint 不退化

### 手动回归
至少覆盖：
- tab 切换
- 广告组合筛选影响主表
- 顶部筛选切换后分页重置
- 从广告组合进入广告活动视图
- 从广告活动进入活动详情页
- 广告日志 tab 只读展示

## 验收标准

满足以下条件即可认为第一版设计达标：
1. 页面只有 8 个业务相关 tab，不包含广告位和 AMC
2. 左侧广告组合筛选能真实影响主表
3. 顶部全局筛选在各 tab 下行为一致
4. tab 切换后只切对象与列，不切整套页面模型
5. 主表支持分页、空态、错误态、加载态
6. 能从广告组合稳定下钻到广告活动
7. 能从广告活动进入详情页
8. Mock API 与前端字段契约一致，不依赖临时兜底拼装

## 实施顺序建议

1. 收敛 `AdManagement.tsx` 为页面 shell
2. 抽离左侧筛选、顶部工具栏、tab 区和表格面板
3. 建立统一 query state 与 URL 同步
4. 实现基于 schema 的表格渲染
5. 对齐下钻与详情跳转链路
6. 补齐前端与 API 契约测试
7. 进行手动回归

## 非目标提醒

本设计不覆盖：
- 真实 Amazon Ads 写入逻辑
- 广告位 / AMC 模块
- 复杂图表分析中台
- 高级自动化预算策略
- 非当前运营流程必需的广告管理能力
